# 技术文档数据集设计笔记

这个文件是我给自己记的数据集设计思路，后面扩数据的时候按这个来。

## 范围先别铺太大

第一阶段我还是想先盯住一个技术文档子领域，不想一开始把很多工具混在一起。范围太散的话，后面分析结果会很乱，也不方便看清楚 retriever 到底哪里有问题。

现在比较适合起步的方向：

- Git 工作流
- Linux 命令行使用
- Docker 部署基础
- PyTorch 训练流程

## 文档 schema

每个 chunk 先保留最基本的几个字段：

- `doc_id`
- `title`
- `text`
- `source`
- `domain`

后面如果数据量起来了，再补这些字段：

- `section_path`
- `tool_name`
- `version`
- `task_type`

这些字段后面做过滤实验和错误归因会方便很多。

## 问题类型不要太单一

我希望问答样本至少覆盖这几类：

- 概念题
- 步骤题
- 参数 / 选项题
- 故障排查题

比如：

- `git rebase` 主要解决什么问题？
- 递归搜索并显示行号时，`grep` 常用哪些选项？
- Docker 数据卷为什么有用？
- PyTorch 中 `DataLoader` 的作用是什么？

## 做分析时优先看什么

后面拿到实验结果后，我会优先看这几个问题：

- 是不是检索阶段就没找到正确 chunk
- 是不是检索到了，但回答阶段没抽出来
- 是不是答案漏掉了关键步骤或关键约束
- 是不是把相似命令、相似参数或者相似工具搞混了

## 错误标签先这么定

- `retrieval_miss`
- `generation_failure`
- `partial_answer`
- `missing_step`
- `option_confusion`
- `tool_confusion`

## 第一版公开出来的目标规模

我觉得第一版如果能做到下面这个规模，就已经够看了：

- 50 到 100 个技术文档 chunk
- 30 到 50 条问答样本
- 2 到 3 个检索基线
- 1 份比较像样的错误分析报告
