# 项目路线图

## 总目标

把当前这个 demo 仓库逐步做成一个适合放在 GitHub 上、能支撑研究生申请介绍的机器学习项目。

## 推荐对外表述

不要把它说成“我做了个 AI 聊天机器人”。

更推荐这样介绍：

`一个面向中文计算机技术文档的 RAG 评测平台，重点关注基准构建、检索器对比和失败案例分析。`

这样的表述更能体现：

- 数据集构建能力
- 实验设计能力
- 结果分析能力
- 系统迭代能力

## 8 周迭代建议

### 第 1 周

- 清理仓库结构
- 写好中文 README
- 明确只做一个技术文档子领域

### 第 2 周

- 收集 50 到 100 篇技术文档页面
- 统一文档 schema
- 构建 30 到 50 条带金标准引用的问答

### 第 3 周

- 改进 chunk 切分策略
- 对比按章节切分和按长度切分
- 记录不同 chunk 大小下的 hit@k

### 第 4 周

- 加入稠密检索
- 对比 BM25 和稠密检索
- 做一张小型 benchmark 表

### 第 5 周

- 加入 reranker 或混合检索
- 分析关键词检索赢的场景
- 分析语义检索赢的场景

### 第 6 周

- 接入真实 LLM
- 强制回答引用检索证据
- 统计幻觉和步骤缺失问题

### 第 7 周

- 扩展评测指标
- 完善失败案例 taxonomy
- 选出 5 个代表性错误案例

### 第 8 周

- 打磨 README 和图表
- 补实验报告
- 准备简历描述和面试讲解稿

## 面试时应该能回答的问题

项目成熟后，你最好能清楚回答下面这些问题：

- 为什么选择技术文档作为领域？
- benchmark 是怎么构建的？
- 为什么某种检索方式更强？
- 当前最主要的错误类型是什么？
- 你准备如何提升事实一致性？

如果这些问题你都能回答得比较自然，这个项目就已经很像“研究准备项目”了，而不是普通课程作业。

- 可复现实验结果
- 按 issue 推进的迭代记录
- 每轮实验后都有简洁的 markdown 报告
